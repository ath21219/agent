### 自前エージェント開発のロードマップ相談

#### まず、ベースプロジェクトの選択について

結論から述べると、**どの既存プロジェクトもベースとしては採用せず、独自のアーキテクチャで構築することを推奨します**。ただし、車輪の再発明は最小限に抑え、各プロジェクトから「部品」として使えるものは積極的に取り込みます。

この判断に至る理由を整理します。

Project AIRIをベースにする場合、VRMサポートは存在しますが、プロジェクト全体がモノレポで4言語（TypeScript/Rust/Python/Go）にまたがり、OODAループやAlayaメモリは現在進行形で大規模な再設計中です。流動するアーキテクチャの上に自前の構築を重ねると、上流の破壊的変更に巻き込まれるリスクが高く、かつ不要な複雑性（Minecraft連携、Factorio連携、ブラウザ内WebGPU推論等）を抱え込むことになります。

Open-LLM-VTuberをベースにする場合、完成度は最高ですが、フロントエンドがLive2D前提で設計されており、VRMフロントエンドを追加するにはフロントエンドの実質的な新規開発が必要です。v2.0が「完全書き直し」の計画段階にあるため、v1のコードベースへの投資が無駄になるリスクもあります。

**したがって、自分のコアとなるアーキテクチャを自前で持ちつつ、各プロジェクトの成果を外部サービス・ライブラリとして活用する構成が最も合理的です。**

#### フェーズ別ロードマップ

##### Phase 0：基盤選定と最小動作確認（1〜2週間）

このフェーズの目標は「VRMモデルがブラウザ上に表示され、LLMと会話でき、声が出る」という最小限のループを回すことです。

技術スタックは以下とします。フロントエンドはTypeScript（React or Vue + Three.js + @pixiv/three-vrm）、バックエンドはPython（FastAPI + WebSocket）です。フロントエンドのフレームワーク選択について、Reactの場合はReact Three Fiberによる宣言的な3D操作の利便性があり、Vueの場合はProject AIRIのコードを参照しやすい利点があります。開発者の経験に合わせて選択してください。

LLM接続には、Project AIRIが開発した**xsAI**（TypeScript向け）またはPython向けの**LiteLLM**を使います。いずれも複数のLLMプロバイダを統一インターフェースで扱えるため、Gemini・Claude・OpenRouter等の切り替えがAPI keyの変更だけで済みます。

TTS/STTには、Project AIRIのサブプロジェクト**unspeech**の利用を検討してください。OpenAI互換の`/audio/speech`と`/audio/transcriptions`エンドポイントを任意のTTS/ASRバックエンドに対してプロキシするサーバーで、バックエンド側のTTS/STT実装を完全に抽象化できます。

VRM表示の参照実装としては、pixivのChatVRMのコードが最もシンプルで理解しやすく、Amicaのコードがリップシンクと感情表情の実装を含むためそこから抽出するのが効率的です。

##### Phase 1：VRM表現の充実（2〜4週間）

Phase 0の骨格に対して、VRMモデルの「生きている感」を作り込みます。

**プロシージャルアニメーション**として、以下の4つを実装します。自動まばたき（ランダムな間隔でBlendShapeの`blink`を駆動、人間の平均まばたき間隔は3〜7秒）、呼吸動作（胸のボーンにサイン波の微小な上下動を適用）、アイドル時の体の揺れ（Perlinノイズによる頭部・上半身の微細な動き）、視線追従（マウスカーソルまたはカメラの検出位置にVRMのLookAtを向ける）。Project AIRIのVRM制御コード（`packages/stage-ui`内）にこれらの実装が既にあるため、参照してください。

**感情駆動の表情制御**では、LLMの出力に感情タグ（`[joy]`、`[sadness]`等）を含めるようシステムプロンプトで指示し、フロントエンドでパースしてBlendShapeに反映します。VRMは標準で`happy`、`angry`、`sad`、`relaxed`、`surprised`のBlendShapeGroupを持っているため、Open-LLM-VTuberのemotionMapと同様のマッピング表を定義するだけで基本的な表情制御が実現できます。重要なのは、表情の遷移をステップ的ではなく補間（lerp）で行うことです。0.3秒程度のイージングを入れるだけで、表情が「切り替わる」のではなく「変化する」印象になります。

**ボディモーション**については、Phase 1では事前定義のアニメーションクリップ（Mixamoなどから取得したうなずき、手振り、体重移動等）をLLMの感情タグに応じて再生する方式で十分です。プロシージャルな全身モーション生成はPhase 3以降の課題とします。

##### Phase 2：音声対話と知覚の統合（2〜4週間）

**音声対話ループ**を完成させます。STT（音声→テキスト）→ LLM（思考）→ TTS（テキスト→音声）→ リップシンク（音声→口の動き）のパイプラインです。Open-LLM-VTuberが実装している「音声割り込み」（エコーキャンセルなしでAI自身の発話を除外する仕組み）は体験品質に大きく影響するため、VAD（Voice Activity Detection）＋自己発話タイムスタンプによるフィルタリングをこのフェーズで実装してください。Silero VADはブラウザ内で動作し、軽量で精度が高いです。

**視覚認知**として、カメラ映像と画面キャプチャをマルチモーダルLLM（Gemini、GPT-4V等）に送信する機能を実装します。Open-LLM-VTuberの実装が参考になります。技術的には、一定間隔（例：5秒ごと）またはイベントトリガー（ユーザーが話しかけた時）でフレームをキャプチャし、Base64エンコードしてLLMのマルチモーダル入力として送信するだけです。

ここで、あなたが提起した**「エージェントが自身のビジュアルモデルがユーザーにどう映っているかを把握する」**という構想についてお答えします。

これは技術的に**段階的に実現可能**です。ただし、各段階で達成できることの性質が異なります。

**第1段階：自己外見の記述的理解（現時点で実現可能）**。エージェントのVRMモデルのスクリーンショット（つまりエージェント自身の「鏡像」）をマルチモーダルLLMに入力し、「これがあなたの見た目です」とシステムプロンプトで伝えます。現在のGPT-4V/Geminiレベルのモデルは画像を詳細に記述できるため、「自分は青い髪の少女で、今笑顔をしている」程度の自己認識は可能です。Josh Whitonの「AIミラーテスト」研究では、5つのマルチモーダルAIのうち4つがこのテストに合格したと報告されています。

**第2段階：自己表現と内部状態の一貫性（中程度の実装で実現可能）**。エージェントが「自分は今悲しい表情をしているはず」と内部状態から推論し、実際にVRMの表情が悲しい状態になっているかをスクリーンショットで確認する、というフィードバックループです。これは内部状態（感情タグ）と外部状態（VRM BlendShape値）を対応付けるテーブルをエージェントのコンテキストに含めることで、マルチモーダル入力なしでも概念的には実現できます。

**第3段階：ユーザー反応の予測（部分的に実現可能、ただし限界あり）**。これは心理学で「心の理論（Theory of Mind）」と呼ばれる能力に相当します。2025年のNature論文での研究によれば、現在のLLMは心の理論テストにおいて人間に匹敵する成績を示す場合がある一方で、その能力は脆弱で文脈依存的です。arxivのToM-agent研究（2025年1月）では、LLMベースのエージェントに心の理論を付与する手法が提案されています。

実装としては、エージェントのシステムプロンプトに「ユーザーモデル」を持たせるアプローチが現実的です。具体的には、「ユーザーは自分の笑顔を見ると安心する傾向がある」「ユーザーは長い沈黙を不安に感じる」といったユーザーの反応パターンを記憶に蓄積し、行動選択時にそれを参照させます。カメラでユーザーの表情を認識し（マルチモーダルLLM入力）、過去の反応パターン（記憶）と照合して「今ユーザーは困惑しているようだ。自分の説明が分かりにくかったかもしれない」と推論する、というフローです。

ただし率直に言えば、**現在のLLMの心の理論は「それらしい推測を生成できる」段階であり、「正確に予測できる」段階ではありません**。この機能はPhase 4以降の実験的取り組みとして位置づけ、まずはユーザーの表情認識→記憶蓄積→プロンプトへの反映、という基盤を作ることを優先するのが現実的です。

##### Phase 3：長期記憶とパーソナリティ（2〜4週間）

**長期記憶**には、**Letta**（旧MemGPT）をそのまま外部サービスとして利用することを推奨します。自前実装すべきでない理由は明確で、LettaはMemGPTの研究論文に基づく「自己編集型メモリ」を完成度高く実装しており、LLM自身がコンテキストウィンドウ内外のデータの移動を管理するという仕組みは、単なるベクトルDB検索よりも根本的に優れた記憶管理を提供するからです。Open-LLM-VTuberでの統合実績もあり、Ollama経由のローカルLLMでもセルフホストで動作します。

Lettaの上に、必要に応じてProject AIRIのAlayaメモリが提案するコンセプト（時間減衰、感情重み付き検索）を追加する層を被せることは、Phase 5以降の発展課題として残します。

**パーソナリティシステム**については、Phase 3の段階ではProject AIRIのソウルコンテナのフル実装ではなく、**より軽量なアプローチ**を取ります。具体的には、パーソナリティ設定をJSON構成ファイルとして定義し（AITuberKitのキャラクター設定ファイルに近い形式）、LLMへのシステムプロンプトの動的構築と、LLM出力の軽量な後処理（冗長性の制御、応答長の制限）を組み合わせます。

ここで**ソウルコンテナの効果に対する疑問**に直接お答えします。

ソウルコンテナのアイデアは設計思想として美しいですが、**その効果の大部分は「よく設計されたシステムプロンプト」と「出力の後処理ルール」でも達成できます**。ソウルコンテナが本当に差別化を発揮するのは、複数のソウルを動的に切り替える必要がある場合（例：同じエージェントが場面に応じて異なる人格を使い分ける場合）や、パーソナリティの一貫性を外部から検証可能にしたい場合（テスト可能性）です。個人用のコンパニオンエージェントであれば、これらの需要が発生するまではプロンプト＋後処理で十分であり、後からソウルコンテナパターンにリファクタリングすることも容易です。

##### Phase 4：「生命らしさ」の追求（継続的）

ここからが既存プロジェクトのどれもまだ十分に達成していない領域であり、あなたの開発が独自の価値を生み出し始めるフェーズです。

**能動的行動**。ユーザーの入力がなくても、エージェントが自発的に行動を起こす仕組みです。Open-LLM-VTuberの「AI proactive speaking」を参考にしつつ、行動のトリガーをより多様にします。例えば、一定時間の沈黙後に「退屈そうにする」、カメラでユーザーが離席したことを検知して「待っている」アニメーションに移行する、時刻に応じて挨拶する、といった行動です。これらはタイマーイベント＋知覚入力＋LLM判断の組み合わせで実現でき、OODAループの本格実装なしでも段階的に導入可能です。

**内面状態の可視化**。Open-LLM-VTuberの「内面思考表示」を参考に、発話されない思考や感情をUIに表示します。これに加えて、VRMモデルの微細な挙動（視線が泳ぐ、手をもじもじする等）で「言語化されない内面」を身体的に表現することで、「この存在には発話以外の内面がある」という感覚を強めます。

**記憶に基づく関係性の変化**。Lettaの長期記憶を活用して、エージェントの態度が時間とともに変化する仕組みです。初対面では丁寧だったのが徐々にくだけてくる、過去に話した話題を自然に参照する、ユーザーの好みを学習して先回りする、といった変化です。これはシステムプロンプト内のユーザーモデル（Lettaの`core_memory`に保存）を定期的に更新することで実装できます。

##### Phase 5以降：高度なアーキテクチャへの段階的移行

Phase 4までで十分に機能するエージェントが動作している状態を確認した上で、必要に応じて以下に取り組みます。

**OODAループ的な思考構造の導入**。Phase 4で導入した能動的行動のトリガーが増え、知覚チャネル（カメラ、画面、音声、時刻等）が複雑化した段階で、線形パイプラインでは管理しきれなくなります。その時点で、知覚の集約（Observe）、文脈との統合（Orient）、行動選択（Decide）、実行（Act）を明示的なフェーズとして分離するリファクタリングを行います。最初から導入するのではなく、複雑性が要求した時点で導入する、というアプローチです。

**Alayaメモリ的な記憶特性の追加**。Lettaの上に時間減衰や感情重み付きのスコアリング層を追加する実験です。これが実際に「生命らしさ」にどの程度寄与するかは、Phase 4の段階で十分な対話データが蓄積されてから評価できます。

#### 「自身のビジュアルがどう映っているか」機能の具体的な実装案

この機能はあなたの構想の中で最もユニークな部分なので、もう少し具体的に掘り下げます。

**実装方法**: エージェントのVRM描画キャンバスから定期的にスクリーンショットを取得し（Three.jsの`renderer.domElement.toDataURL()`）、これを「鏡像入力」としてエージェントのコンテキストに含めます。システムプロンプトには「この画像はあなた自身の現在の見た目です」と記述します。

**活用シナリオ**: エージェントが「今自分は怒った表情をしてしまっている。ユーザーを怖がらせたかもしれないから、意識的に表情を和らげよう」と判断し、BlendShapeの変更を自発的に指示する、というフローが可能になります。これは行動選択のフィードバックループとして機能し、内部状態→外見→認知→修正、という自己調整サイクルを生み出します。

**実用的な注意点**: マルチモーダルLLM呼び出しのたびに画像を送信するとコストとレイテンシが増大するため、この「鏡像確認」は常時ではなく、表情変更後や重要なインタラクションの前後に限定するのが現実的です。

Phase 2 のテーマは**「対話品質の深化と外部連携の基盤」**です。以下の優先度で提案します。

**Phase 2-A（2〜3 日）— LLM のプロンプト改善 + コンテキスト管理強化**
現在の会話履歴はメモリ内の配列で、ブラウザリロードで消えます。また、システムプロンプトの感情タグ指示は最低限のものです。会話の質を上げるため、システムプロンプトの強化（キャラクター設定の深化、応答スタイルの調整）、会話履歴の localStorage 永続化、トークン数管理（古いメッセージの要約・削除）を行います。感情タグの出力安定性も改善します。これはコード変更量が少なく、体験への影響が大きい項目です。

**Phase 2-B（3〜5 日）— TTS の完全ストリーミング再生**
現在は文単位で TTS API を呼び、AudioBuffer 全体をデコードしてから再生しています。これを OpenAI TTS の PCM ストリーミング出力（`response_format: pcm`）を使い、チャンクが到着するたびに Web Audio API の AudioWorklet で逐次再生する構成に改善します。最初の音声が聞こえるまでの時間をさらに 0.3〜0.5 秒短縮できます。

**Phase 2-C（1〜2 日）— API キーのサーバーサイド移行**
LLM（ストリーミング対応）と TTS の API Route プロキシを作成し、`NEXT_PUBLIC_` キーを撤廃します。Phase 2-B でストリーミングを実装した後の方が、プロキシの設計が一度で済むため効率的です。

**Phase 2-D（3〜5 日）— カメラ / 画面キャプチャ視覚認知**
ユーザーのカメラ映像や画面共有をキャプチャし、Gemini の Vision API に送ることで「今何をしているか」を理解させる機能です。これにより「画面に表示されているコードについて質問する」「ユーザーの表情を読み取る」等が可能になります。

**Phase 2-E（5〜7 日）— Letta / MemGPT メモリ統合**
長期記憶の基盤です。Python バックエンド（FastAPI + Letta）を導入し、会話のサマリーや重要な事実をベクトル DB に蓄積します。「先週話したあの件」のような参照が可能になります。
